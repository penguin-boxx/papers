\section{Introduction}
Escape analysis determines whether a value can outlive its lexical scope. It originated as a compiler optimization, for example in stack allocation and the removal of intermediate objects~\cite{park1992escape,blanchet1999escape}. In modern languages the same question is increasingly part of the type discipline: Rust ties it to lifetimes and borrowing~\cite{matsakis2014rust}, OCaml develops a related idea through a modal treatment of locality~\cite{lorenzen2024oxidizing}, recently extended to data-race freedom~\cite{georges2025data}, and Kotlin uses contracts for restricted forms of non-local control~\cite{akhin2021kotlin}. Verifying such mechanisms has become a topic in its own right: RustBelt provides a semantic foundation for the safety of Rust~\cite{jung2018rustbelt,jung2020stacked}, while Prusti, Creusot, RustHornBelt, and RefinedRust lift properties of Rust programs to machine-checked specifications~\cite{astrauskas2022prusti,denis2022creusot,matsushita2022rusthornbelt,gaher2024refinedrust}.

A second reason to make escape analysis type-based comes from algebraic effect handlers. Handlers replace a whole zoo of built-in control effects---exceptions, mutable state, generators, coroutines, asynchrony---with a single construct: a program \emph{performs} an operation and the nearest enclosing handler interprets it, with access to the captured continuation~\cite{plotkin2013handling}. We carry this economy into the type system as well. In a capability-based system an operation is invoked through a capability value; if that capability leaves the dynamic extent of its handler, the program would call a handler that no longer exists. Effekt, capture tracking, and evidence-passing handlers address this by different means~\cite{brachthauser2020effects,brachthauser2022effects,boruch2023capturing,xie2021generalized,xie2022first}, usually through explicit capture sets or a substantial polymorphic infrastructure. The $Core_\Delta$ calculus~\cite{stoyan2025existential} offers a coarser but compact alternative: lifetimes form a two-point lattice $\free \sqsubseteq \local$ with an explicit minimum, and compound data types hide the lifetimes of their fields existentially, recovering them on pattern matching. Its design, subtyping, and a prototype type-checker are presented in that companion paper; the present work contributes the Rocq/Coq mechanization, the effect-handler extension with a marker-indexed semantics, and the machine-checked safety and escape guarantees.

This combination is convenient for the programmer but delicate for the metatheory. Existential lifetimes interact with substitution, subtyping, and variance; handlers add continuation capture and runtime markers. A pen-and-paper proof can easily miss a leak: a $\local$ value returned inside a closure's environment, or a capability captured by a resumption that outlives its handler. The goal of this work is a machine-checked metatheory of $Core_\Delta$ extended with effect handlers, stating exactly which escape-analysis guarantees follow from the typing rules.

\paragraph{Model assumptions.}
Three assumptions keep the model concise and machine-checkable. First, we work purely at the type level: there is no heap, store, or mutable reference, so ``escape'' is a lexical-lifetime discipline and stack allocation, ownership, and region placement are motivating applications rather than memory properties we prove. Second, lifetimes form the two-point lattice $\free \sqsubseteq \local$; multi-level regions are a natural generalization. Third, each effect declares one operation. Within these bounds the result is a faithful, executable, and checkable specification.

\paragraph{Contributions.}
The organizing principle is \emph{economy of mechanism}: handlers are the single effect construct, and they add no escape-analysis structure \emph{to the type system}---only runtime machinery (markers and a scoping invariant). First, we extend $Core_\Delta$ with single-operation, capability-passing effect handlers and give a small-step semantics with marker-indexed evaluation contexts and reified continuations (Section~\ref{sec:calculus}). Second, we mechanize the entire system in Rocq/Coq with pure de Bruijn indices: syntax, lifetime and type subtyping, typing---including the constructor \emph{minimum} rule and the variance-aware elimination of existential lifetimes at \texttt{match}---and operational semantics. Third, we prove progress, preservation, and \emph{unconditional} type safety: every runtime invariant of the marker semantics is proved preserved by reduction (Section~\ref{sec:mechanization}). Fourth, we establish three escape-analysis guarantees as machine-checked statements about the \emph{dynamics}, all following for source programs from the initial typing alone: along every reduction sequence a runtime capability stays \emph{confined} to its handler (\texttt{source\_capability\_never\_exposed}), every value crossing a handler boundary, in either direction, is typed at an escapable type (\texttt{source\_handler\_boundary\_noloc}), and a program typed at a free data type only ever delivers values with a non-local lifetime annotation (\texttt{source\_local\_value\_does\_not\_escape}). Fifth, we provide an executable example library and machine-checked \emph{negative} witnesses that reject locality-violating programs (Section~\ref{sec:evaluation}).

\section{Related Work and Problem Statement}\label{sec:related-work}
Classical region systems made lifetime an explicit object of typing long before modern ownership languages~\cite{tofte1997region,fluet2006monadic}. Their aim is close to ours: to replace a global analysis with local conditions at function boundaries. The difference is that $Core_\Delta$ does not parameterize every user type by a lifetime. A constructor computes the minimum of its arguments' lifetimes, and pattern matching introduces fresh existential variables bounded by the scrutinee's lifetime. This reduces annotation but complicates the proof of type preservation under reduction.

Effect handlers have been studied denotationally and operationally~\cite{plotkin2013handling,zyuzin2021contextual,van2022handling,van2024framework,tang2025modal}; the practical line includes their integration into OCaml~\cite{sivaramakrishnan2021retrofitting} and evidence-passing compilation~\cite{xie2021generalized}. Substructural disciplines at the handler boundary are an active line of work: linear types made sound for multi-shot handlers~\cite{tang2024soundly} and affine tracking of continuation use~\cite{vanrooij2025affect} police \emph{how} continuations are used, where $Core_\Delta$ instead polices \emph{which} values may cross the boundary. Most relevant are capability-based systems: Effekt provides a type discipline for scope- and type-based reasoning~\cite{brachthauser2020effects,brachthauser2022effects}, while capture tracking refines function types with sets of captured values~\cite{boruch2023capturing}. $Core_\Delta$ records coarser information: a value is either free to escape or locally confined. In contrast to systems that extend the type language with effect rows or capture sets~\cite{brachthauser2020effects,boruch2023capturing}, it adds no effect-specific type structure: effects reuse the data and function formers, so the machinery that bounds value escape also confines capabilities, at the cost of one operation per effect.

Capability confinement has been pursued by several routes. Lexically scoped handlers confine operations by syntactic scope rather than by a runtime marker discipline~\cite{wu2014effect,biernacki2019binders}; capability-based exception and effect systems for Scala track provenance through the typing context~\cite{odersky2021safer,odersky2022scoped}, and mainstream languages are adding comparable context-passing facilities~\cite{KEEP-context-parameters}. Second-class values forbid escape through a use-site restriction close in spirit to our $\local$ lifetime~\cite{osvald2016gentrification}; row-typed effect systems instead thread an effect row through every type~\cite{leijen2017type}; and evidence- or tunnel-based designs route control without rows~\cite{zhang2016accepting,zhang2020handling}. Recent mechanizations of capture calculi pursue a related goal with explicit boxes~\cite{xu2023formalizing,xu2025s}, and a broader survey situates these lines~\cite{stoyan2024modern}. $Core_\Delta$ differs by folding capability confinement into the same two-point value-lifetime discipline it already uses for ordinary data, adding no effect-specific type structure.

Mechanizing the metatheory of type systems builds on the POPLMark challenge and the engineering of formal metatheory~\cite{aydemir2005poplmark,aydemir2008engineering}. Ott and Autosubst show that syntax and substitution can be automated~\cite{sewell2010ott,schaefer2015autosubst,stark2019autosubst2}. We deliberately use pure de Bruijn indices with ordinary recursive shift and substitution functions: the definitions are transparent and portable, at the cost of an explicit substitution metatheory.

The model must therefore (i) express lifetimes in types without overloading the user with parameters; (ii) confine a runtime capability to its handler; (iii) admit first-class continuations without letting them escape; and (iv) have a checkable metatheory suitable as a compiler specification.

\section{The $Core_\Delta$ Calculus}\label{sec:calculus}
The mechanization covers the system in full; the figures of this section present the core rules, with routine well-formedness, length, and effect-lookup premises elided.

\subsection{Background: algebraic effect handlers}
Readers unfamiliar with effect handlers may read them as a generalization of exception handling. An \emph{effect} declares an \emph{operation}: a request that a piece of code can make of its surrounding context. The code raises the request with $\keyword{perform}$, and the nearest enclosing $\keyword{handle}$ block interprets it. The crucial difference from $\keyword{throw}$/$\keyword{catch}$ is that a handler need not merely abort the suspended computation: it also receives that computation's \emph{continuation}---the rest of the work, captured up to the handler---as a first-class value, the \emph{resumption}, and may run it. Concretely, in
\[
\keyword{handle}\ \{\,\dots\;\keyword{perform}\ \mathit{ask}\,()\;\dots\,\}\ \keyword{with}\ \mathit{ask}()\Rightarrow k.\ \keyword{resume}\ k\ 42,
\]
each $\keyword{perform}\ \mathit{ask}()$ suspends the running computation and hands the handler its continuation $k$; the handler replies with $\keyword{resume}\ k\ 42$, restarting $k$ with the value $42$, so every $\mathit{ask}()$ evaluates to $42$. A handler that returns without resuming recovers exception-like early exit; resuming exactly once recovers reader, state, and logging effects; resuming more than once expresses backtracking and generators. One $\keyword{handle}$/$\keyword{perform}$/$\keyword{resume}$ mechanism thus covers a wide range of control effects.

We use a \emph{capability-passing} presentation: entering a handler mints a \emph{capability}---a value that grants permission to invoke the operation---and passes it into the handled code, where $\keyword{perform}$ takes it as its receiver. A capability is meaningful only while its handler is on the stack: once control leaves the $\keyword{handle}$ block the handler is gone, so a capability that escaped would have no handler to interpret it. Preventing that escape is precisely the analysis this calculus performs---the same discipline it applies to ordinary local values, now applied to capabilities.

\subsection{Syntax}
Figure~\ref{fig:syntax} gives the grammar. Lifetimes are the two-point lattice with a minimum: $\Delta_1+\Delta_2$ denotes the join in $\free \sqsubseteq \local$, which operationally is the \emph{more restrictive} (shorter) of the two---$\local$ as soon as either operand is $\local$. A function type $\tau \arr{\Delta} \sigma$ carries a \emph{closure lifetime} $\Delta$; a data type $K\,\Delta\,\overline{\tau}$ is annotated with the lifetime for which a $K$-value may be used. Effect capabilities reuse the data machinery: a capability for effect $E$ has type $E\,\local\,\overline{\tau}$. Core functions are unary; multi-argument functions and surface effect declarations are sugar; type and lifetime abstractions are prenex---in the artifact every $\Lambda$-chain ends in a $\lambda$. Three runtime-only forms arise during reduction: the capability value $\cap_E^{m,\sigma}$, the delimiter $\hdlr_m$, and the reified resumption $\reic_m$, each carrying a runtime marker $m$; type annotations ($\sigma$ is the handler's public answer type) are recorded so that preservation can re-type the reducts, and the mechanized source $\keyword{handle}$ and $\keyword{perform}$ forms carry answer- and result-type annotations (elided here) for the same reason. Of the three, only the capability and the delimiter are primitive constructors of the artifact: $\reic_m^{\,\tau,\sigma}(x.\,t)$ abbreviates the ordinary abstraction $\lambda(x{:}A).\,\hdlr_m^{\,\tau,\sigma}\,t$---$A$ being the resumption's argument type---that re-installs the delimiter, so the rules \rname{T-Resume} and \rname{H-Resume} of Figure~\ref{fig:handlers} are derived rather than primitive.

\subsection{Typing}
Figure~\ref{fig:typing} shows the core typing rules. The escape discipline is concentrated in three places.

\emph{Functions (\rname{T-Lam}).} A $\lambda$ receives a closure lifetime $\Delta$ that must bound $\cl_\Gamma(t)$, the \emph{capture lifetime} of its body. $\cl_\Gamma(t)$ is $\local$ if the body syntactically contains a runtime capability; otherwise it is the minimum ($+$) of the lifetimes $\ltof_\Gamma$ of the captured free variables' types. Thus a function that captures a $\local$ value (or a capability) necessarily has closure lifetime $\local$, and by \rname{S-Fun} and $\ltof$ it can no longer be embedded where a $\free$ lifetime is required. This is how a local value is kept from crossing a function boundary, without any separate ``result must be non-local'' side condition.

\emph{Constructors (\rname{T-Ctor}).} A constructor's result lifetime must dominate the minimum of its field lifetimes: $\lsub{\Gamma}{\ltof(\overline{\rho})}{\ltof(K\,\Delta\,\overline{\tau})}$, where $\overline{\rho}$ are the instantiated field types. A value built from fields is therefore safe for no longer than its shortest-lived field. A disjointness side condition keeps data tags apart from effect tags and from the distinguished $\Any$ tag.

\emph{Pattern matching (\rname{T-Match}).} Matching a scrutinee of type $K\,\Delta\,\overline{\tau}$ introduces $n$ fresh lifetime variables $\overline{l}$ bounded by $\Delta$, types the success branch under the recovered field types to a type $\eta$, and then \emph{eliminates} the fresh variables from $\eta$ with $\elimp_{\overline{l}\mapsto\Delta}$: a variable in a positive position is replaced by its bound $\Delta$, in a negative position by $\free$, and an occurrence in an invariant position is \emph{rejected}. This recovers the existentially hidden lifetimes soundly and is exactly the rule that blocks leakage through containers and functions.

\subsection{Handlers: typing and semantics}
A capability value has a $\local$ data type, and a resumption is a function with closure lifetime $\local$ (Figure~\ref{fig:handlers}, top): both are therefore non-escaping by the function discipline above. The source rule \rname{T-Handle} types the operation body $t_{op}$ against the public answer type $T_R$, with the operation argument $x{:}\sigma_\beta$ and the resumption $k{:}\rho_\beta\arr{\local}T_R$ in scope, and types the handled body $b$ against $T_B$ with the capability $c$ in scope; $\sigma_\beta,\rho_\beta$ are the operation's declared argument/result types instantiated at the effect's type arguments $\overline{\tau}$. The runtime forms mirror this: a capability records $T_R$ and types its stored operation body the same way (\rname{T-Cap}), and a handler delimiter is transparent to typing, requiring only that the body answer type $T_B$ be non-local and a subtype of the public answer $T_R$ (\rname{T-HandlerM}). The \rname{T-Perform} rule instantiates the operation signature once more with the perform-site type arguments $\overline{S}$, giving argument type $\sigma_{\overline{S}}$ and result type $\rho_{\overline{S}}$; it requires the argument type and each $S_i$ to be non-local---$\noloc_\Gamma(\tau)$ abbreviates $\lsub{\Gamma}{\ltof_\Gamma(\tau)}{\free}$---so a value crossing the operation boundary cannot smuggle a local capability out of scope. The operation \emph{result}, however, may be local, which lets a deep resumption deliver local values back to the operation site.

The operational rules (Figure~\ref{fig:handlers}, bottom) are layered: head reductions $t \hstep t'$ have no congruence, and $t \step t'$ closes them under well-formed call-by-value evaluation contexts $\mathcal{E}$ via $\rname{S-Step}$. Entering a handler ($\rname{S-Handle}$) picks a fresh marker $m$, builds the capability $\cap_E^{m}$, substitutes it into the body, and installs the delimiter $\hdlr_m$. A $\keyword{perform}$ on a capability with marker $m$, sitting inside a pure (delimiter-free for $m$) context $P$ under the matching $\hdlr_m$, captures $P$ as a resumption $\reic_m$ that re-installs the delimiter, and runs the operation body on the argument and that resumption ($\rname{H-Perform}$). Applying a resumption re-installs the delimiter around the resumed body ($\rname{H-Resume}$); the handlers are thus \emph{deep}. The invariant $\mathsf{well\_scoped}\;\overline{m}\;t$ records that every runtime capability with marker $m$ lies under its $\hdlr_m$ or within the active set $\overline{m}$, together with a provenance condition on stored operation bodies (Section~\ref{sec:mechanization}); source programs satisfy $\mathsf{well\_scoped}\;[\,]\;t$.

\textit{Economy of mechanism.} Handlers add no escape-analysis type structure. The type grammar of Figure~\ref{fig:syntax} is closed under the same formers with or without effects---there is no effect row, capture set, or effect-lifetime parameter. An effect is registered as a context signature and denoted by a reserved tag on the data former $K\,\Delta\,\overline{\tau}$, kept disjoint from data tags by the side condition $\mathsf{lookupEff}(\Gamma,K)=\bot$ on \rname{T-Ctor}/\rname{T-Match}; a capability is then a data value at lifetime $\local$ and a resumption a $\local$ function. Confinement and value non-escape are therefore governed by one lattice $\free\sqsubseteq\local$ and one elimination discipline, and the reader, state, exception, and optionality handlers of Section~\ref{sec:evaluation} are four instances of this one construct. What handlers \emph{do} add is operational---runtime markers, the delimiter $\hdlr_m$, and the $\mathsf{well\_scoped}$ invariant; the economy is at the level of types and escape analysis, not the runtime. Because a capability is a $\local$ data value and a resumption a $\local$ function, confining a capability to its handler is an instance of value non-escape rather than a separate device; the cost is coarse $\free$/$\local$ information and one operation per effect.

\subsection{Safety theorems}
We write $\step$ for the small step; theorems are stated over a typing environment $\Gamma$ containing only runtime bindings (\texttt{eval\_ctx} in the artifact), which covers the empty environment of a closed program. The soundness proof tracks runtime invariants alongside typing, bundled with the typing judgment as \texttt{safety\_invariants} in the artifact: $\mathsf{marker\_types\_safe}\;t$ (all delimiter and resumption answer-type annotations for one marker agree), $\mathsf{marker\_annots\_closed}\;t$ (a technical strengthening that keeps those annotations stable under substitution), $\mathsf{well\_scoped}\;[\,]\;t$ (capabilities stay under their delimiters, strengthened with marker \emph{provenance}: each capability's stored operation body is well-scoped at the scope outside that capability's own delimiter), and $\mathsf{rt\_closed}\;t$ (stored operation bodies are term-closed). All hold vacuously for source programs, which carry no runtime markers, and all are proved preserved by reduction; progress needs only $\mathsf{well\_scoped}$ and $\mathsf{marker\_types\_safe}$.

\emph{Progress.} If $\tj{\Gamma}{t}{\tau}$, $\mathsf{well\_scoped}\;[\,]\;t$ and $\mathsf{marker\_types\_safe}\;t$, then $t$ is a value or $t\step t'$ for some $t'$. The proof uses a stronger open form (\texttt{progress\_open\_safe}) that, for a non-empty active marker set, additionally allows a pending $\keyword{perform}$ on an in-scope capability; at the empty set this third case is impossible.

\emph{Preservation.} If $\tj{\Gamma}{t}{\tau}$ and $t\step t'$, then $\tj{\Gamma}{t'}{\tau}$. Ordinary redexes use the corresponding substitution-preservation lemmas; the runtime forms $\cap$, $\hdlr_m$, $\reic_m$ are introduced only by the handler rules and immediately typed at lifetime $\local$.

\emph{Type safety.} A well-typed program never reaches a \emph{stuck} term ($\neg\,\mathsf{value}$ and no step). The capstone \texttt{type\_soundness} is \emph{unconditional}: the invariant bundle above is proved preserved along every reduction sequence---the key lemma being preservation of $\mathsf{well\_scoped}$ by reduction (\texttt{step\_preserves\_well\_scoped})---so for a source program safety follows from the initial typing alone (\texttt{source\_type\_soundness}).

\emph{Confinement.} \texttt{capability\_confined}: under the empty marker set a runtime capability $\cap_E^{m,\sigma}$ can never occupy a pure (delimiter-free for $m$) position---it must sit under its delimiter $\hdlr_m$. Transported along the dynamics, a capability reachable from a well-typed program is never exposed outside its handler: the transported form (\texttt{capability\_never\_exposed}) takes $\mathsf{well\_scoped}$ along the trace as a premise, and for source programs that premise is discharged by the preserved invariant bundle (\texttt{source\_capability\_never\_exposed}), leaving only the initial typing. Confinement is therefore a proved theorem, not an assumption.

\emph{Boundary impermeability.} \texttt{source\_handler\_boundary\_noloc}: along any execution of a well-typed source program, every value crossing a handler boundary---the operation argument entering at a $\keyword{perform}$, or the body result leaving through the delimiter $\hdlr_m$---is typed at an escapable type ($\noloc_\Gamma$); when that type is an ordinary data type (not an effect tag and not $\Any$), the value is a constructor whose top-level lifetime annotation is non-local (\texttt{source\_boundary\_value\_non\_local}). The one $\local$ value that does cross is the resumption itself, and its non-escape is exactly the confinement guarantee above: together the two theorems make the handler boundary dynamically impermeable to local values in both directions.

\emph{Non-escape.} \texttt{local\_value\_does\_not\_escape}: if a computation stays typed at a free data type $K\,\free\,\overline{\tau}$ along every reduction and terminates at a value $v$, then $v$ is a constructor whose top-level lifetime annotation is provably non-local. By preservation the trace premise again disappears for source programs (\texttt{source\_local\_value\_does\_not\_escape}): a well-typed marker-free program typed at $K\,\free\,\overline{\tau}$ can only ever deliver a non-local value. A subtyping-level companion, \texttt{local\_data\_not\_escapes}, shows the coercion $\tsub{\Gamma}{K\,\local\,\overline{\tau}}{K\,\free\,\overline{\tau}}$ underivable in any runtime environment: local data cannot even be retyped as free, let alone delivered. The type information therefore forbids the observable leakage of local values.

\subsection{Trusted base and proof engineering}
The substitution metatheory---shift and substitution commutation laws, weakening, narrowing, inversions---and \emph{all} per-redex substitution-preservation lemmas (for $\beta$, type-, and lifetime-application, \texttt{match}, and \texttt{perform}), which earlier revisions isolated as assumptions, are proved outright, as is the preservation of every runtime invariant. The development contains no \texttt{Axiom}, no \texttt{Admitted}, and no \texttt{Conjecture}: \texttt{Print Assumptions} on every theorem of this section---including the capstone \texttt{type\_soundness} and the source-program corollaries---reports \emph{closed under the global context}, so the trusted base is the Rocq kernel and its standard library.

The well-scopedness invariant deserves a proof-engineering comment. An early revision assumed a marker-confinement \emph{axiom} that turned out to be false as stated; it was replaced by the explicit structural invariant $\mathsf{well\_scoped}$, first carried as a hypothesis of the capstone theorem and finally proved preserved by reduction. The step that made the preservation proof go through was reformulating the invariant \emph{without} a marker-freshness requirement: the final predicate records \emph{provenance}---each capability's stored operation body must be well-scoped at the scope outside that capability's own delimiter, exactly what the $\keyword{perform}$ contraction needs---and is monotone under scope extension, so installing a new delimiter around a term or moving it under binders by substitution cannot invalidate it. The operational semantics still mints globally fresh markers in \rname{S-Handle}; what turned out to be unnecessary is carrying freshness in the \emph{preserved invariant}: $\beta$-substitution duplicates subterms and so destroys uniqueness, but the semantics always addresses the \emph{innermost} occurrence of a marker, so shadowing is harmless. The development is about $2.4\times10^{4}$ lines with about $9\times10^{2}$ \texttt{Qed} obligations; the substitution tier alone accounts for roughly $58\%$ of the lines, the price of pure de Bruijn indices with hand-rolled shift and substitution functions.

\section{Evaluation}\label{sec:evaluation}
The evaluation has three parts.

\emph{Reproducibility.} The verified core builds with standard Rocq/Coq and the Stdlib only; \texttt{make} generates a makefile from \texttt{src/\_CoqProject} and compiles the modules of Table~\ref{tab:modules}.

\emph{Positive examples.} \texttt{Examples.v}/\texttt{ExamplesProofs.v} contain $Core_\Delta$ programs and handlers for reader, state, exception, and optionality, each with a machine-checked typing proof and a reduction trace. Listing~\ref{lst:reader} shows the reader handler in surface notation: the effect $\mathit{Reader}$ declares one operation $\mathit{ask}$, and the handler answers every $\keyword{perform}$ by resuming with the fixed environment. The capability \texttt{c} has the $\local$ data type $\mathit{Reader}\,\local\,\mathsf{Int}$ and the resumption has closure lifetime $\local$, so by \rname{T-Lam} any function capturing either is itself $\local$ and cannot flow where a $\free$ value is expected. The final function \texttt{leak} is rejected: its $\keyword{handle}$ body answers at the type $\mathit{Reader}\,\local\,\mathsf{Int}$, and the premise $\noloc_\Gamma(T_B)$ of \rname{T-Handle} fails. The boundary guarantee is also instantiated concretely on this program: \texttt{readerExample\_boundary\_noloc} specializes \texttt{source\_handler\_boundary\_noloc} to its execution.

\emph{Negative witnesses.} Three locality violations are rejected by \emph{machine-checked} computations over the decidable locality checks $\noloc$ and $\elimp$ (proved by \texttt{reflexivity}): storing a $\local$ reader inside a $\free$ \texttt{Option} (\texttt{rejected\_testWithState}: $\noloc(\mathtt{Option}\,\free\,(\mathit{Reader}\,\local\,\mathsf{Unit}))=\mathsf{false}$); using an existential lifetime contravariantly (\texttt{rejected\_crashEndo}: $\elimp$ returns \emph{none}); and exposing \texttt{Option}\,$\local$ through a $\free$ interface (\texttt{rejected\_crashBox}). These witness that the analysis rejects the canonical leakage channels, not merely that it accepts good programs. Rejection at typing time is the static complement of the dynamic guarantees of Section~\ref{sec:mechanization}: programs that would leak do not typecheck, and programs that typecheck provably do not leak. The first witness merits a closer look. The offending program stores a reader capability in an \texttt{Option} and hands it out at the escapable interface type $\mathtt{Option}\,\free\,(\mathit{Reader}\,\local\,\mathsf{Unit})$. The check computes the lifetime restriction $\ltof$ of that type as the minimum over all lifetime annotations occurring in it; the $\local$ of the stored capability type drags the whole minimum down to $\local$, and the judgment $\lsub{\Gamma}{\local}{\free}$ that $\noloc$ would then need is exactly the underivable statement behind \texttt{local\_data\_not\_escapes} (Section~\ref{sec:mechanization}). The rejection is thus a $\mathsf{false}$ computed by \texttt{reflexivity}---a \texttt{Qed}-checked fact, not a hand-waved counterexample.

A practical use is the design of a compiler intermediate language where stack or region allocation must coexist with capability-safe handlers: the typing rules give local checks at function boundaries, and the semantics shows which runtime invariant a correct capture/resume implementation must maintain.

\section{Limitations and Future Work}\label{sec:discussion}
The metatheory is closed for the calculus as stated---no axioms, admitted proofs, or conditional theorems remain---so the limitations are those of the calculus itself. First, effects are single-operation, the price of the economy-of-mechanism design: because an effect reuses the data former rather than a bespoke effect type, it carries one operation signature. Multi-operation effects are encoded by a sum-typed argument---the example library's state handler dispatches its \texttt{get}/\texttt{put} commands through one operation this way---but the surface notation becomes cumbersome, and operation tags are the natural extension. Second, operation typing is asymmetric: the argument and the perform-site type instantiations must be non-local, while the result may be local (so resumptions can deliver local values back to the operation site); lifting the remaining argument restriction would require threading the argument lifetime through the continuation type. Third, there is no memory model: connecting the lexical discipline to actual stack or region allocation, and widening the two-point lattice to multi-level regions, are the main semantic extensions. Finally, the typing relation is specified but not yet realized as a verified algorithm; the locality side conditions ($\noloc$, $\elimp$, $\mathsf{is\_value}$) are already decidable functions, so extracting a certified reference checker is a concrete next step.

\section{Conclusion}\label{sec:conclusion}
We presented a machine-checked metatheory of $Core_\Delta$ extended with single-operation algebraic effect handlers, mechanizing the full system---syntax, lifetime and type subtyping, typing with a constructor minimum rule and variance-aware elimination of existential lifetimes, and a marker-indexed small-step semantics---and proving progress, preservation, and unconditional type safety, with capability confinement, handler-boundary impermeability, and operational non-escape of local values following for source programs from the initial typing alone. An executable example library and machine-checked negative witnesses show the analysis on reader/state/exception/optionality handlers and on programs it must reject. The artifact is useful as a checked specification for compilers that combine safe value placement with first-class, capability-safe effect handlers; adding multi-operation effects, widening the lifetime lattice to multi-level regions, and extracting a certified checker are the natural next steps.
