\section{Introduction}
Escape analysis determines whether a value can outlive its lexical scope. It originated as a compiler optimization, for example for stack allocation and the removal of intermediate objects~\cite{park1992escape,blanchet1999escape}. In modern languages the same question is increasingly part of the type discipline: Rust ties it to lifetimes and borrowing~\cite{matsakis2014rust}, OCaml develops a related idea through a modal treatment of locality~\cite{lorenzen2024oxidizing}, and Kotlin uses contracts for restricted forms of non-local control~\cite{akhin2021kotlin}. Verifying such mechanisms has become a topic in its own right: RustBelt provides a semantic foundation for the safety of Rust~\cite{jung2018rustbelt,jung2020stacked}, while Prusti, Creusot, RustHornBelt, and RefinedRust lift properties of Rust programs to machine-checked specifications~\cite{astrauskas2022prusti,denis2022creusot,matsushita2022rusthornbelt,gaher2024refinedrust}.

A second reason to make escape analysis type-based comes from algebraic effect handlers. Handlers replace a whole zoo of built-in control effects---exceptions, mutable state, generators, coroutines, asynchrony---with a single construct: a program \emph{performs} an operation and the nearest enclosing handler interprets it, with access to the captured continuation~\cite{plotkin2013handling}. We carry this economy into the type system as well. In a capability-based system an operation is invoked through a capability value; if that capability leaves the dynamic extent of its handler, the program would call a handler that no longer exists. Effekt, capture tracking, and evidence-passing handlers address this by different means~\cite{brachthauser2020effects,brachthauser2022effects,boruch2023capturing,xie2021generalized,xie2022first}, usually through explicit capture sets or a substantial polymorphic infrastructure. The $Core_\Delta$ calculus~\cite{stoyan2025existential} offers a coarser but compact alternative: lifetimes form a two-point lattice $\free \sqsubseteq \local$ with an explicit minimum, and compound data types hide the lifetimes of their fields existentially, recovering them on pattern matching.

This combination is convenient for the programmer but delicate for the metatheory. Existential lifetimes interact with substitution, subtyping, and variance; handlers add continuation capture and runtime markers. A pen-and-paper proof can easily overlook a case in which a local value or a capability ends up in a function's result. The goal of this work is a machine-checked metatheory of $Core_\Delta$ extended with effect handlers, stating exactly which escape-analysis guarantees follow from the typing rules.

\paragraph{Scope and non-goals.}
To keep the claims precise we fix what is and is not modeled. The calculus has \emph{no} heap, store, or mutable references: ``escape'' here is a \emph{type-level lexical-lifetime discipline}, and stack allocation, ownership, and region placement are motivating \emph{applications} rather than properties we prove about memory. The lifetime lattice is deliberately the two-point lattice $\free \sqsubseteq \local$; richer multi-level regions are out of scope. Each effect declares exactly one operation. Within these bounds the contribution is a faithful, executable, and checkable specification rather than a full language.

\paragraph{Contributions.}
The organizing principle is \emph{economy of mechanism}: handlers are the single effect construct, and they add no escape-analysis type structure. First, we extend $Core_\Delta$ with single-operation, capability-passing effect handlers and give a small-step semantics with marker-indexed evaluation contexts and reified continuations (Section~\ref{sec:calculus}). Second, we mechanize the entire system in Rocq/Coq with pure de Bruijn indices: syntax, lifetime and type subtyping, typing---including the constructor \emph{minimum} rule and the variance-aware elimination of existential lifetimes at \texttt{match}---and operational semantics. Third, we prove progress (under two runtime invariants), single-step preservation, and type safety (Section~\ref{sec:mechanization}). Fourth, we establish two escape-analysis guarantees: a runtime capability is \emph{confined} to its handler, and a value observed at a free data type \emph{does not escape} a top-level $\local$. Fifth, we provide an executable example library and machine-checked \emph{negative} witnesses that reject locality-violating programs (Section~\ref{sec:evaluation}). Because a capability is a data value at lifetime $\local$ and a resumption a $\local$ function, confining a capability to its handler is an instance of value non-escape rather than a separate device, and the type system gains no effect rows, capture sets, or effect-lifetime parameters; the cost is coarse $\free$/$\local$ information and one operation per effect.

\section{Related Work and Problem Statement}\label{sec:related-work}
Classical region systems made lifetime an explicit object of typing long before modern ownership languages~\cite{tofte1997region,fluet2006monadic}. Their aim is close to ours: replace a global analysis with local conditions at function boundaries. The difference is that $Core_\Delta$ does not parameterize every user type by a lifetime. A constructor computes the minimum of its arguments' lifetimes, and pattern matching introduces fresh existential variables bounded by the scrutinee's lifetime. This reduces annotation but complicates the proof of type preservation under reduction.

Effect handlers have been studied denotationally and operationally~\cite{plotkin2013handling,zyuzin2021contextual,van2022handling,van2024framework,tang2025modal}; the practical line includes their integration into OCaml~\cite{sivaramakrishnan2021retrofitting} and evidence-passing compilation~\cite{xie2021generalized}. Most relevant are capability-based systems: Effekt provides a type discipline for scope- and type-based reasoning~\cite{brachthauser2020effects,brachthauser2022effects}, while capture tracking refines function types with sets of captured values~\cite{boruch2023capturing}. $Core_\Delta$ records coarser information: a value is either free to escape or locally confined. In contrast to systems that extend the type language with effect rows or capture sets~\cite{brachthauser2020effects,boruch2023capturing}, it adds no effect-specific type structure: effects reuse the data and function formers, so the machinery that bounds value escape also confines capabilities, at the cost of one operation per effect.

Mechanizing the metatheory of type systems builds on POPLMark and the engineering of formal metatheory~\cite{aydemir2005poplmark,aydemir2008engineering}. Ott and Autosubst show that syntax and substitution can be automated~\cite{sewell2010ott,schaefer2015autosubst,stark2019autosubst2}. We deliberately use pure de Bruijn indices with ordinary recursive shift and substitution functions: the definitions are transparent and portable, at the cost of an explicit substitution metatheory.

The model must therefore (i) express lifetimes in types without overloading the user with parameters; (ii) confine a runtime capability to its handler; (iii) admit first-class continuations without letting them escape; and (iv) have a checkable metatheory suitable as a compiler specification.

\section{The $Core_\Delta$ Calculus}\label{sec:calculus}

\subsection{Background: algebraic effect handlers}
Readers unfamiliar with effect handlers may read them as a generalization of exception handling. An \emph{effect} declares an \emph{operation}: a request that a piece of code can make of its surrounding context. The code raises the request with $\keyword{perform}$, and the nearest enclosing $\keyword{handle}$ block interprets it. The crucial difference from $\keyword{throw}$/$\keyword{catch}$ is that a handler need not merely abort the suspended computation: it also receives that computation's \emph{continuation}---the rest of the work, captured up to the handler---as a first-class value, the \emph{resumption}, and may run it. Concretely, in
\[
\keyword{handle}\ \{\,\dots\;\keyword{perform}\ \mathit{ask}\,()\;\dots\,\}\ \keyword{with}\ \mathit{ask}()\Rightarrow k.\ \keyword{resume}\ k\ 42,
\]
each $\keyword{perform}\ \mathit{ask}()$ suspends the running computation and hands the handler its continuation $k$; the handler replies with $\keyword{resume}\ k\ 42$, restarting $k$ with the value $42$, so every $\mathit{ask}()$ evaluates to $42$. A handler that returns without resuming recovers exception-like early exit; resuming exactly once recovers reader, state, and logging effects; resuming zero or many times expresses backtracking and generators. One $\keyword{handle}$/$\keyword{perform}$/$\keyword{resume}$ mechanism thus covers a wide range of control effects.

We use a \emph{capability-passing} presentation: entering a handler mints a \emph{capability}---a value that grants permission to invoke the operation---and passes it into the handled code, where $\keyword{perform}$ takes it as its receiver. A capability is meaningful only while its handler is on the stack: once control leaves the $\keyword{handle}$ block the handler is gone, so a capability that escaped would have nothing to interpret it. Preventing that escape is precisely the analysis this calculus performs---the same discipline it applies to ordinary local values, now applied to capabilities.

\subsection{Syntax}
Figure~\ref{fig:syntax} gives the grammar. Lifetimes are the two-point lattice with a minimum: $\Delta_1+\Delta_2$ denotes the join in $\free \sqsubseteq \local$, which operationally is the \emph{more restrictive} (shorter) of the two---$\local$ as soon as either operand is $\local$. A function type $\tau \arr{\Delta} \sigma$ carries a \emph{closure lifetime} $\Delta$; a data type $K\,\Delta\,\overline{\tau}$ is annotated with the lifetime for which a $K$-value may be used. Effect capabilities reuse the data machinery: a capability for effect $E$ has type $E\,\local\,\overline{\tau}$. Core functions are unary; multi-argument functions and surface effect declarations are sugar. Three runtime-only forms arise during reduction: the capability value $\cap_E^{m}$, the delimiter $\hdlr_m$, and the reified resumption $\reic_m$, each carrying a runtime marker $m$.

\subsection{Typing}
Figure~\ref{fig:typing} shows the core typing rules. The escape discipline is concentrated in three places.

\emph{Functions (\rname{T-Lam}).} A $\lambda$ receives a closure lifetime $\Delta$ that must bound $\cl_\Gamma(t)$, the \emph{capture lifetime} of its body. $\cl_\Gamma(t)$ is $\local$ if the body syntactically contains a runtime capability; otherwise it is the minimum ($+$) of the lifetimes $\ltof_\Gamma$ of the captured free variables' types. Thus a function that captures a $\local$ value (or a capability) necessarily has closure lifetime $\local$, and by \rname{S-Fun} and $\ltof$ it can no longer be embedded where a $\free$ lifetime is required. This is how a local value is kept from crossing a function boundary, without any separate ``result must be non-local'' side condition.

\emph{Constructors (\rname{T-Ctor}).} A constructor's result lifetime must dominate the minimum of its field lifetimes: $\lsub{\Gamma}{\ltof(\overline{\rho})}{\ltof(K\,\Delta\,\overline{\tau})}$, where $\overline{\rho}$ are the instantiated field types. A value built from fields is therefore safe for no longer than its shortest-lived field. A disjointness side condition keeps data tags apart from effect tags.

\emph{Pattern matching (\rname{T-Match}).} Matching a scrutinee of type $K\,\Delta\,\overline{\tau}$ introduces $n$ fresh lifetime variables $\overline{l}$ bounded by $\Delta$, types the success branch under the recovered field types to a type $\eta$, and then \emph{eliminates} the fresh variables from $\eta$ with $\elimp_{\overline{l}\mapsto\Delta}$: a variable in a positive position is replaced by its bound $\Delta$, in a negative position by $\free$, and an occurrence in an invariant position is \emph{rejected}. This recovers the existentially hidden lifetimes soundly and is exactly the rule that blocks leakage through containers and functions.

\subsection{Handlers: typing and semantics}
A capability value has a $\local$ data type, and a resumption is a function with closure lifetime $\local$ (Figure~\ref{fig:handlers}, top): both are therefore non-escaping by the function discipline above. A handler delimiter is transparent to typing, requiring only that the body answer type $T_B$ be non-local and a subtype of the public answer $T_R$. The \rname{T-Perform} rule requires the operation argument to be non-local ($\noloc_\Gamma$), so a value crossing the operation boundary cannot smuggle a local capability out of scope; the operation \emph{result}, however, may be local, which lets a deep resumption deliver local values back to the operation site.

The operational rules (Figure~\ref{fig:handlers}, bottom) are layered: head reductions $t \hstep t'$ have no congruence, and $t \step t'$ closes them under well-formed call-by-value evaluation contexts $E$ via $\rname{S-Step}$. Entering a handler ($\rname{S-Handle}$) picks a fresh marker $m$, builds the capability $\cap_E^{m}$, substitutes it into the body, and installs the delimiter $\hdlr_m$. A $\keyword{perform}$ on a capability with marker $m$, sitting inside a pure (delimiter-free for $m$) context $P$ under the matching $\hdlr_m$, captures $P$ as a resumption $\reic_m$ that re-installs the delimiter, and runs the operation body on the argument and that resumption ($\rname{H-Perform}$). Applying a resumption re-installs the delimiter around the resumed body ($\rname{H-Resume}$); the handlers are thus \emph{deep}. The invariant $\mathsf{marker\_ok}\;\overline{m}\;t$ records that every runtime capability with marker $m$ lies under its $\hdlr_m$ or within the active set $\overline{m}$; source programs satisfy $\mathsf{marker\_ok}\;[\,]\;t$.

\textit{Economy of mechanism.} Handlers add no escape-analysis type structure. The type grammar of Figure~\ref{fig:syntax} is closed under the same formers with or without effects---there is no effect row, capture set, or effect-lifetime parameter. An effect is registered as a context signature and denoted by a reserved tag on the data former $K\,\Delta\,\overline{\tau}$, kept disjoint from data tags by the side condition $\mathsf{lookupEff}(\Gamma,K)=\bot$ on \rname{T-Ctor}/\rname{T-Match}; a capability is then a data value at lifetime $\local$ and a resumption a $\local$ function. Confinement and value non-escape are therefore governed by one lattice $\free\sqsubseteq\local$ and one elimination discipline, and the reader, state, exception, and optionality handlers of Section~\ref{sec:evaluation} are four instances of this one construct. What handlers \emph{do} add is operational---runtime markers, the delimiter $\hdlr_m$, and the $\mathsf{marker\_ok}$ invariant; the economy is at the level of types and escape analysis, not the runtime.

\subsection{Safety theorems}
We write $\step$ for the small step and $\Gamma$ for an evaluation context (only runtime bindings). Two runtime invariants accompany typing: $\mathsf{marker\_ok}\;[\,]\;t$ (capabilities stay under their delimiters) and $\mathsf{marker\_types\_safe}\;t$ (all delimiter/resumption answer-type annotations for one marker agree).

\emph{Progress.} If $\tj{\Gamma}{t}{\tau}$, $\mathsf{marker\_ok}\;[\,]\;t$ and $\mathsf{marker\_types\_safe}\;t$, then $t$ is a value or $t\step t'$ for some $t'$. The proof uses a stronger open form (\texttt{progress\_open\_safe}) that, for a non-empty active marker set, additionally allows a pending $\keyword{perform}$ on an in-scope capability; at the empty set this third case is impossible.

\emph{Preservation.} If $\tj{\Gamma}{t}{\tau}$ and $t\step t'$, then $\tj{\Gamma}{t'}{\tau}$. Ordinary redexes use the corresponding substitution-preservation lemmas; the runtime forms $\cap$, $\hdlr_m$, $\reic_m$ are introduced only by the handler rules and immediately typed at lifetime $\local$.

\emph{Type safety.} A well-typed program satisfying both invariants never reaches a \emph{stuck} term ($\neg\,\mathsf{value}$ and no step). This is \texttt{type\_safety}/\texttt{type\_soundness}: the invariants are preserved along every reduction sequence and a state satisfying them is never stuck.

\emph{Confinement.} The marker invariant, together with \texttt{marker\_ok\_plug\_cap\_pure\_in}, ensures a runtime capability $\cap_E^{m}$ never occupies a pure (delimiter-free for $m$) position outside its $\hdlr_m$; under the empty marker set no capability occurs at all. Confinement is thus a consequence of the preserved invariant rather than a separate axiom.

\emph{Non-escape.} \texttt{local\_value\_does\_not\_escape}: if a computation stays typed at a free data type $K\,\free\,\overline{\tau}$ along every reduction and terminates at a value $v$, then $v$ is a constructor whose top-level lifetime annotation is provably non-local. The type information therefore forbids the observable leakage of local values.

\subsection{Trusted base}
The substitution metatheory---shift and substitution commutation laws, weakening, narrowing, and inversion lemmas, together with type- and lifetime-substitution preservation---is proved in the \texttt{subst/} package (about $1.6\times10^{4}$ lines); the constructor case of type-substitution preservation has been discharged, leaving only its \texttt{match} case. The remaining obligations form a small, explicit interface: the term-substitution lemma and its parallel form, the \texttt{match} case of type substitution, and two marker-confinement facts; the per-redex preservation steps for the five redex forms follow from these. The development is about $2.5\times10^{4}$ lines with on the order of $10^{3}$ \texttt{Qed} obligations.

\section{Evaluation}\label{sec:evaluation}
The evaluation has three parts.

\emph{Reproducibility.} The verified core builds with standard Rocq/Coq and the Stdlib only; \texttt{make} generates a makefile from \texttt{src/\_CoqProject} and compiles the modules of Table~\ref{tab:modules}.

\emph{Positive examples.} \texttt{Examples.v}/\texttt{ExamplesProofs.v} contain $Core_\Delta$ programs and handlers for reader, state, exception, and optionality, each with a machine-checked typing proof and a reduction trace. A schematic reader handler illustrates the shape: an effect $\mathit{Reader}$ with one operation $\mathit{ask}:\mathsf{Unit}\to e$ is run by
$\keyword{handle}\;\mathit{Reader}\,[e]\,\{\mathit{ask}\Rightarrow \reic(\mathit{env})\}\;\keyword{in}\;\dots\keyword{perform}\;\mathit{cap}\;()\dots$,
where each $\keyword{perform}$ resumes with the fixed environment $\mathit{env}$; the capability and resumption are $\local$, so neither can be returned by the handled block.

\emph{Negative witnesses.} Three locality violations are rejected by \emph{machine-checked} computations over the decidable locality checks $\noloc$ and $\elimp$ (proved by \texttt{reflexivity}): storing a $\local$ reader inside a $\free$ \texttt{Option} (\texttt{rejected\_testWithState}: $\noloc(\mathtt{Option}\,\free\,(\mathit{Reader}\,\local\,\mathsf{Unit}))=\mathsf{false}$); using an existential lifetime contravariantly (\texttt{rejected\_crashEndo}: $\elimp$ returns \emph{none}); and exposing \texttt{Option}\,$\local$ through a $\free$ interface (\texttt{rejected\_crashBox}). These witness that the analysis rejects the canonical leakage channels, not merely that it accepts good programs.

A practical use is the design of a compiler intermediate language where stack or region allocation must coexist with capability-safe handlers: the typing rules give local checks at function boundaries, and the semantics shows which runtime invariant a correct capture/resume implementation must maintain.

\section{Limitations and Future Work}\label{sec:discussion}
The substitution-preservation interface (five per-redex lemmas and two marker facts) is isolated for proof engineering; keeping it small and explicit is deliberate, and its members are the obvious targets for any future refactor toward an automated substitution infrastructure such as Autosubst. A second limitation is single-operation effects, the price of the economy-of-mechanism design: because an effect reuses the data former rather than a bespoke effect type, it carries one operation signature. Multi-operation effects are encoded by a sum-typed argument, but the surface notation becomes cumbersome, and operation tags are the natural extension. Third, operation typing is asymmetric: the argument must be non-local, while the result may now be local (so resumptions can deliver local values back to the operation site); lifting the remaining argument restriction would require threading the argument lifetime through the continuation type. Fourth, there is no memory model: connecting the lexical discipline to actual stack or region allocation, and widening the two-point lattice to multi-level regions, are the main semantic extensions. Finally, the typing relation is specified but not yet realized as a verified algorithm; the locality side conditions ($\noloc$, $\elimp$, $\mathsf{is\_value}$) are already decidable functions, so extracting a certified reference checker is a concrete next step.

\section{Conclusion}\label{sec:conclusion}
We presented a machine-checked metatheory of $Core_\Delta$ extended with single-operation algebraic effect handlers, giving the full formal system---syntax, lifetime and type subtyping, typing with a constructor minimum rule and variance-aware elimination of existential lifetimes, and a marker-indexed small-step semantics---and proving progress, preservation, type safety, capability confinement, and operational non-escape of local values. An executable example library and machine-checked negative witnesses show the analysis on reader/state/exception/optionality handlers and on programs it must reject. The artifact is useful as a checked specification for compilers that combine safe value placement with first-class, capability-safe effect handlers; completing the substitution-preservation interface, adding multi-operation effects, and extracting a certified checker are the natural next steps.
